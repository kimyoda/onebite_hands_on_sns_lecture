## section03_quiz

### 1. React ContextAPI가 전역 상태 관리에 가지는 주요 한계점은

- **A. 상태 변경 시 불필요한 전체 컴포넌트 리렌더링**
- B. 복잡한 데이터 구조를 지원하지 않는다 -> Context는 객체, 배열 등 어떤 값이든 전달이 가능하다
- C. 비동기 작업 처리가 어렵다 -> 비동기는 Context 자체 문제가 아니라 설계 문제다
- D. 라이브러리 크기가 너무 크다 -> Context는 React 내장 기능이라 추가 용량이 없다

* 해설: Context API는 특정 값 변경 시 해당 Context를 사용하는 모든 컴포넌트를 리렌더링할 수 있다. 불필요한 성능 저하를 야기할 수 있어 전역상태 관리에 적합하지 않을 수 있다.

- Context API는 React에 내장된 상태 공유 도구이다. `Provider`로 감싼 하위 컴포넌트 어디서든 값을 꺼내 쓸 수 있으나, Context 값이 바뀌면 해당 Context를 구독하는 모든 컴포넌트가 리렌더링이 된다.

### 2. Zustand에서 상태를 업데이트 하는 가장 권장되는 방법은?

- A. `get().property = value` 직접 할당 -> Zustand 상태는 직접 변경 불가 (immer 없이)
- **B. `set((state) => ({ property: sate.property + 1}))` 함수형 업데이트**
- C. `this.setState()` 사용 -> 클래스 컴포넌트 문법, Zustand와 무관
- D. 새로운 `create` 함수 호출 -> 스토어 자체를 새로 만드는 것, 상태 업데이트가 아니다.

* 해설: `set` 메서드에 콜백 함수를 전달하는 함수형 업데이트는 최신 상태를 기반으로 안전하게 상태를 변경한다.

- Zustand `set`은 두 가지 방식으로 쓸 수 있다.

```ts
// 방식 1: 직접 값 지정 (현재 상태에 의존) X
set({ count: get().count + 1 });

// 방식 2: 함수형 업데이트 (항상 최신 상태를 보장)
set((state) => ({ count: state.count + 1 }));
```

- 함수형이 안전한 이유

```ts
const increament = async () => {
  await someAsyncWork();

  set((state) => ({ count: state.count + 1 })); // 항상 최신값
};
```

### 3. Zustand 셀렉터의 리렌더링 최적화

- A. 상태 변경 시 모든 컴포넌트 리렌더링 방지 -> 셀렉터로 선택한 값이 바귀면 리렌더링은 발생한다
- **B. 스토어의 특정 값만 선택하여 해당 값 변경 시에만 리렌더링**
- C. 리렌더링 주기를 수동으로 제어 -> 셀렉터는 자동으로 작동하고 수동 제어가 아니다
- D. 스토어에서 데이터를 캐싱하여 리렌더링 최소화 -> 캐싱은 React Query 등의 역할, 셀렉터는 구독 범위 한정

* 해설: 셀렉터 함수는 스토어에서 필요한 데이터만 추출하게 해주고, 추출된 값이 변경될 때만 컴포넌트를 리렌더링한다. 이로 인해 불필요한 리렌더링을 효과적으로 줄일 수 있다.

- 셀렉터 없이 스토어 전체를 구독하면, 관계없는 값이 바뀌어도 리렌더링 된다.

```ts
const useTodosStore = create(combine({
  todos: [],
  filter: "all",
  username: "홍길동"
}, ...));

// 이 컴포넌트는 username이 바뀌어도 리렌더링 안 됨
function TodoList() {
  const todos = useTodosStore((state) => state.todos); // todos만 구독
  return <>{todos.map(...)}</>;
}
```

### 4. Zustand의 `immer` 미들웨어는 상태 업데이트를 어떻게 단순화하나?

- A. 비동기 상태 관리를 자동 처리 -> 비동기는 별도 미들웨어(redux-thunk 등)의 역할
- **B. 불변성 관리를 위해 상태를 직접 구문 허용**
- C. 스토어 상태를 자동으로 로컬 스토리지에 저장 -> 이는 `persist` 미들웨어의 역할
- D. 여러 스토어를 하나로 결합 -> 이는 `combine` 미들웨어의 역할

* 해설: `immer` 미들웨어는 불변성을 유지하면서 마치 상태를 직접 수정하는 것처럼 코드를 작성할수 있게 해준다. 복잡한 중첩 객체 업데이트를 간결하게 만든다.

- immer 동작원리: immer는 Proxy 객체로 상태를 감싸서, 변경 내용을 추적 한 뒤 새로운 불변 객체를 생성한다. 직접 변경처럼 보이지만 실제론 불변성을 지킨다.
- React/Zustand는 기본적으로 불변성(immutability)을 요구한다. 상태를 직접 수정하면 변경을 감지하지 못한다.

```ts
// immer 없이 중첩 객체 업데이트
set((state) => ({
  user: {
    ...state.user,
    address: {
      ...state.user.address,
      city: "서울",
    },
  },
}));

// immer 사용 (직접 수정하는 것처럼 간결하게)
set((state) => {
  state.user.address.city = "서울"; // 내부적으로는 불변성 유지됨
});

// 배열도 push/splice 등 직접 사용 가능
set((state) => {
  state.todos.push({ id: 1, content: "새 할일" });
});
```

### 5. `persist`미들웨어 사용 시, 액션 함수가 기본적으로 저장되지 않는 이유는?

- A. 액션 함수는 스토어의 일부가 아니기 때문 -> actions도 스토어 객체 안에 있다
- **B. 함수는 JSON으로 직렬화할 수 없기 때문**
- C. 보안상의 이유로 저장 금지 -> 보안과 무관, 순전히 직렬화 문제
- D. 불필요한 스토리지 공간 낭비 방지 -> 의도적 설계가 아니라 JSON의 기술적 한계

* 해설: `persist` 미들웨어는 상태를 JSON 형태로 저장, JSON으로 직렬화할 수 없는 함수는 저장하지 안흔다. 따라서 `serialize` 옵션으로 상태만 저장하는 것이 좋다.

- `persist`는 스토어 상태를 `localStorage` 등에 저장할때 JSONstringify를 사용한다.

```ts
// JSON.stringify로 함수를 직렬화
JSON.stringify({ count: 1, increament: () => {} });
```

- 주의사항

```ts
const useStore = create(
  persist(
    combine(
      { todos: [], filter: "all" },
      (set) => ({
        actions: {
          addTodo: (content: string) => set(...), // 저장 안 됨 (함수)
        }
      })
    ),
    {
      name: "todo-storage",
      // 함수를 제외하고 저장할 상태만 명시적으로 지정하는 것이 좋음
      partialize: (state) => ({ todos: state.todos }),
    }
  )
);
```

- 저장 가능: 문자열, 숫자, 불리언, 배열, 일반객체
- 저장 불가능: 함수 (actions), undefined, Symbol, Map, Set(기본 설정)
