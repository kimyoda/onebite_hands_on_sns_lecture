## 3.1 전역상태관리와 Zustand

### 1. 전역 상태 관리

- 전역 상태 (Global State)
- 컴포넌트의 상태는 기본적으로 자기자신
- React에서 `useState`로 만든 상태는 해당 컴포넌트 안에서만 존재한다.
- 실제 앱에서 여러 컴포넌트가 같은 데이터를 공유해야 하는 경우가 생긴다
  - 사용자 인증 정보(로그인한 유저 이름)
  - 테마 정보(다크모드 설정 등)
  - 장바구니 정보(상품 목록 페이지, 헤더, 결제 페이자가 같은 장바구니 데이터)

### 2. Props Drilling

- Props Drilling 이슈
- 부모 컴포넌트의 상태를 자식에게 props로 내려줘야 한다., 자식의 자식, 자식에게

```
<App>                         ← 유저 정보 보유
  └─ <Layout>                 ← props로 전달
       └─ <Sidebar>           ← props로 전달
            └─ <UserProfile>  ← 여기서 실제로 사용!
```

- Props Drilling이라 한다.
- 중간에 있는 `Layout`, `Sidebar`는 `UserProfile`에 전달하기 위해 쓰지도 않는 props를 받아 아래로 내려보낸다.
- 컴포넌트가 결합도가 높아지고, 유지보수가 어려워져, 코드가 지저분해진다.
- 데이터 흐름을 추적하기도 힘들다.

### 3. 전역 상태 관리를 돕는 도구들

- Context API
- Provider로 감싸면 그 아래 어떤 컴포넌트든 데이터를 바로 꺼내 쓸 수 있다.

```tsx
// Context 생성
const UserContext = createContext(null);

// Provider로 감싸기
<UserContext.Provider value={user}>
  <App />
</UserContext.Provider>;

// 어디서든
const user = useContext(UserContext);
```

- 전역 상태 관리 기능보단 Props Drilling 이슈를 해결하기 위해 제공하는 도구
- 범용적인 전역 상태 관리보다 국소적인 데이터 공유를 위해 더 자주 사용된다

  > Context의 value가 변경되면, 해당 Context를 구독하는 모든 컴포넌트가 리렌더링 된다.

- 국소적인 데이터 공유 목적으로 더 자주 활용된다.

### Zustand를 쓰는 이유

- Redux, Zojotai, Zustand, Recoil 등 라이브러리

- Zustand는 Redux대비 70배 정도 가볍다. 이 차이는 사용자 경험에 꽤 주요하게 영향을 끼친다.
- Zustand를 선택하는 이유
- 가볍다 — 91.5KB, 번들 사이즈에 부담 없음
- 직관적이다 — 복잡한 보일러플레이트 없이 바로 사용 가능
- 많은 사람이 쓴다 — 커뮤니티와 레퍼런스가 풍부
- 성능이 좋다 — 필요한 상태만 구독해 불필요한 리렌더링 방지
- Provider가 필요 없다 — 앱을 감쌀 필요 없이 어디서든 바로 사용

### Zustand 개념(간략)

- `crate` 함수다.

```tsx
import { create } from 'zustand'

const useCountStoer = create((set) => {
  count: 0,
  increase: () => set((state) => state + 1);
  decrease: () => set((state) => state - 1);
});
```

- `create` 함수 안에 상태(state)와 상태를 바꾸는 함수(action)를 함께 정의한다.
- `set`을 호출하면 상태가 업데이트 되고, store를 구독하는 컴포넌트가 자동으로 리렌더링된다.
