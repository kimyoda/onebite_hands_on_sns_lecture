## section03_ai_quiz

### 1. TanStack Query에서 데이터 변경 후 캐시를 무효화는 목적은?

- TanStack Query는 서버에서 가져온 데이터를 캐시(Cache)에 저장해둔다. 덕분에 매번 서버에 요청하지 않아도 빠르게 데이터를 보여줄 수 있다.
- 데이터를 생성/수정/삭제하면, 캐시에 저장된 데이터는 낡은 정보(stale)가 된다.
- 캐시를 무효화(invaildate)하면 TanStackQuery가 서버에서 최신 데이터를 다시 가져와 캐시를 갱신한다.

```js
const queryClient = useQueryClient();

const mutation = useMutation({
  mutationFn: addTodo,
  onSuccess: () => {
    // 할일 추가 후 → todos 캐시 무효화 → 최신 목록 자동 refetch
    queryClient.invalidateQueries({ queryKey: ["todos"] });
  },
});
```

흐름:

```
데이터 변경(mutation)
     ↓
캐시 무효화(invalidateQueries)
     ↓
서버에서 최신 데이터 refetch
     ↓
UI에 최신 데이터 반영 → 데이터 일관성 유지 ✅
```

A. 서버 부하 줄이기 -> 캐시 무효화는 서버에 새로운 요청을 발생, 서버 부하를 줄이는 것과 반대 방향이다. 캐시를 최대한 활용해야 한다
**B. 데이터 일관성 유지하기** -> 캐시 무효화의 이유다. 서버의 실제 데이터와 클라이언트 캐시의 데이터를 일치, 사용자가 항상 정확한 최신 정보를 볼 수 있게 한다.
C. 네트워크 요청 최소화 -> 캐시를 유지하는 것의 목적이다. 무효화는 반대로 네트워크 요청을 발생시킨다.
D. 사용자 세션 관리 -> 세션 관리는 인증(Authentication)과 관련된 영역, 캐시 무효화와는 다른 개념이다.

- 데이터를 변경한 후 캐시를 무효화자 읺으면, 사용자에게 이미 삭제되었거나 수정된 데이터가 그대로 보이는 불일치 현상 발생
- 캐시 무효화는 이를 방지, 서버와 UI의 데이터 일관성을 보장하기 위해 사용

### 2. 서버 응답 전에 사용자 인터페이스를 미리 업데이트하는 기법은 무엇일까요?

- 서버 요청은 시간이 걸린다. 대기 시간동안 UI가 반응이 없으면 답답함을 느낀다. 이를 해결하는 두 가지 전략이 있다.

| 전략                | 설명                              | UX                                  |
| ------------------- | --------------------------------- | ----------------------------------- |
| **비관적 업데이트** | 서버 응답을 기다린 후 UI 업데이트 | 안전하지만 느리게 느껴짐            |
| **낙관적 업데이트** | 서버 응답 전에 UI를 먼저 업데이트 | 빠르게 느껴지지만 실패 시 롤백 필요 |

- 낙관적 업데이트 흐름:

```
사용자가 좋아요 버튼 클릭
     ↓
UI 즉시 업데이트 (하트 ❤️로 변경)  ← 낙관적 업데이트
     ↓
서버에 요청 전송
     ↓
성공 → 그대로 유지 ✅
실패 → 이전 상태(빈 하트 🤍)로 롤백 ↩️
```

```js
useMutation({
  mutationFn: toggleLike,
  onMutate: async () => {
    const previous = queryClient.getQueryData(["post"]);
    queryClient.setQueryData(["post"], (old) => ({ ...old, liked: true })); // 낙관적 업데이트
    return { previous }; // 롤백용 이전 데이터 저장
  },
  onError: (err, _, context) => {
    queryClient.setQueryData(["post"], context.previous); // 실패 시 롤백
  },
});
```

A. 지연 업데이트 -> 존재하지 않는 개념
**B. 낙관적 업데이트** -> 요청이 성공할 것이라고 낙관하고 UI를 먼저 바꾸는 기법. 실패 시 이전 상태로 롤백하여 데이터 정합성을 유지한다.
C. 비간적 업데이트 -> 반대 개념. 요청이 실패할 수도 있다고 비관, 서버 응답을 받은 후에만 UI를 업데이트한다. 안전하지만 느리게 느껴진다.
D. 점진적 업데이트 -> 일반적으로 데이터를 나눠 순차적으로 보여주는 방식(페이지네이션, 스켈레톤 UI 등), 서버 응답 전 업데이트 와는 다르다.

- 낙관적 업데이트는 서버 응답을 기다리지 않고 UI를 즉시 반영, 사용자 경험을 향상시키는 기법이다.
- 실패 상황에 대비해 롤백 로직을 함께 구현해야 한다.
