## section03_ai_quiz

### 1. TanStack Query에서 데이터 변경 후 캐시를 무효화는 목적은?

- TanStack Query는 서버에서 가져온 데이터를 캐시(Cache)에 저장해둔다. 덕분에 매번 서버에 요청하지 않아도 빠르게 데이터를 보여줄 수 있다.
- 데이터를 생성/수정/삭제하면, 캐시에 저장된 데이터는 낡은 정보(stale)가 된다.
- 캐시를 무효화(invaildate)하면 TanStackQuery가 서버에서 최신 데이터를 다시 가져와 캐시를 갱신한다.

```js
const queryClient = useQueryClient();

const mutation = useMutation({
  mutationFn: addTodo,
  onSuccess: () => {
    // 할일 추가 후 → todos 캐시 무효화 → 최신 목록 자동 refetch
    queryClient.invalidateQueries({ queryKey: ["todos"] });
  },
});
```

흐름:

```
데이터 변경(mutation)
     ↓
캐시 무효화(invalidateQueries)
     ↓
서버에서 최신 데이터 refetch
     ↓
UI에 최신 데이터 반영 → 데이터 일관성 유지 ✅
```

A. 서버 부하 줄이기 -> 캐시 무효화는 서버에 새로운 요청을 발생, 서버 부하를 줄이는 것과 반대 방향이다. 캐시를 최대한 활용해야 한다
**B. 데이터 일관성 유지하기** -> 캐시 무효화의 이유다. 서버의 실제 데이터와 클라이언트 캐시의 데이터를 일치, 사용자가 항상 정확한 최신 정보를 볼 수 있게 한다.
C. 네트워크 요청 최소화 -> 캐시를 유지하는 것의 목적이다. 무효화는 반대로 네트워크 요청을 발생시킨다.
D. 사용자 세션 관리 -> 세션 관리는 인증(Authentication)과 관련된 영역, 캐시 무효화와는 다른 개념이다.

- 데이터를 변경한 후 캐시를 무효화자 읺으면, 사용자에게 이미 삭제되었거나 수정된 데이터가 그대로 보이는 불일치 현상 발생
- 캐시 무효화는 이를 방지, 서버와 UI의 데이터 일관성을 보장하기 위해 사용

### 2. 서버 응답 전에 사용자 인터페이스를 미리 업데이트하는 기법은 무엇일까요?

- 서버 요청은 시간이 걸린다. 대기 시간동안 UI가 반응이 없으면 답답함을 느낀다. 이를 해결하는 두 가지 전략이 있다.

| 전략                | 설명                              | UX                                  |
| ------------------- | --------------------------------- | ----------------------------------- |
| **비관적 업데이트** | 서버 응답을 기다린 후 UI 업데이트 | 안전하지만 느리게 느껴짐            |
| **낙관적 업데이트** | 서버 응답 전에 UI를 먼저 업데이트 | 빠르게 느껴지지만 실패 시 롤백 필요 |

- 낙관적 업데이트 흐름:

```
사용자가 좋아요 버튼 클릭
     ↓
UI 즉시 업데이트 (하트 ❤️로 변경)  ← 낙관적 업데이트
     ↓
서버에 요청 전송
     ↓
성공 → 그대로 유지 ✅
실패 → 이전 상태(빈 하트 🤍)로 롤백 ↩️
```

```js
useMutation({
  mutationFn: toggleLike,
  onMutate: async () => {
    const previous = queryClient.getQueryData(["post"]);
    queryClient.setQueryData(["post"], (old) => ({ ...old, liked: true })); // 낙관적 업데이트
    return { previous }; // 롤백용 이전 데이터 저장
  },
  onError: (err, _, context) => {
    queryClient.setQueryData(["post"], context.previous); // 실패 시 롤백
  },
});
```

A. 지연 업데이트 -> 존재하지 않는 개념
**B. 낙관적 업데이트** -> 요청이 성공할 것이라고 낙관하고 UI를 먼저 바꾸는 기법. 실패 시 이전 상태로 롤백하여 데이터 정합성을 유지한다.
C. 비간적 업데이트 -> 반대 개념. 요청이 실패할 수도 있다고 비관, 서버 응답을 받은 후에만 UI를 업데이트한다. 안전하지만 느리게 느껴진다.
D. 점진적 업데이트 -> 일반적으로 데이터를 나눠 순차적으로 보여주는 방식(페이지네이션, 스켈레톤 UI 등), 서버 응답 전 업데이트 와는 다르다.

- 낙관적 업데이트는 서버 응답을 기다리지 않고 UI를 즉시 반영, 사용자 경험을 향상시키는 기법이다.
- 실패 상황에 대비해 롤백 로직을 함께 구현해야 한다.

### 3. 캐시 정규화는 애플리케이션 내 중복 데이터 관리와 어떤 관련이 있을까요?

- 캐시 정규화(Cache Normalization)란, 같은 데이터가 여러 개의 쿼리 캐시에 중복 저장되지 않도록 단일 출처(Single Source of Truth)로 구조화하는 방법이다.

* 정규화 없는 경우 (중복 발생):

```
['posts']          → [{ id: 1, title: '제목', author: '홍길동' }, ...]
['posts', 1]       → { id: 1, title: '제목', author: '홍길동' }  ← 중복!
['user', 'posts']  → [{ id: 1, title: '제목', author: '홍길동' }]  ← 중복!
```

-> id:1 데이터가 수정되면 세 곳을 모두 업데이트해야한다.

- 정규화 적용 후:

```
['posts', 1]  → { id: 1, title: '제목', author: '홍길동' }  ← 단 하나의 원본
다른 캐시들은 id: 1 을 참조  → 항상 최신 데이터 반영
```

-> 한 곳만 업데이트하면 모든 곳에 자동 반영

- 정규화를 통해 중복을 제거, 데이터 업데이트 시 모든 컴포넌트가 동일한 최신 데이터를 참조하게 되어 일관성이 보장된다.

A. 데이터 복구 속도 향상 -> 캐시 정규화의 목적이 아니다. 데이터 복구는 백업/복원 전략과 관련된 개념이다.
**B. 데이터 일관성 유지(정답)** - 정규화의 핵심이다. 중복 데이터를 제거하고 단일 출처로 관리, 여러 컴포넌트가 같은 데이터를 참조할 때 항상 동일한 최신 상태를 유지할 수 있다.
C. 캐시 저장 공간 증가 -> 정반대이다. 중복을 제거하여 저장 공간을 절약(정규화)
D. 보안 취약점 가소 -> 캐시 정규화와 연관이 없다.

- 동일한 데이터를 하나의 캐시 항목으로 단일화, 중복을 제거
- 항상 동일하고 일관된 데이터를 보장한다.

### 4. 시스템에 접근하려는 사용자의 신원을 확인하는 과정을 무엇이라 부를까요?

- 보안에서 자주 혼동되는 두 개념을 명확히 구분해야 한다.

| 구분     | 인증(Authentication)            | 인가(Authorization)                |
| -------- | ------------------------------- | ---------------------------------- |
| 핵심질문 | 당신이 누구입니까?              | 당신은 무엇을 할 수 있습니까?      |
| 목적     | 신원 확인                       | 권한 부여                          |
| 예시     | 로그인, 비밀번호 입력, 지문인식 | 관리자 페이지 접근, 파일 삭제 권한 |
| 순서     | 먼저 수행                       | 인증이후에 수행                    |

- 웹 서비스 예시:

```
로그인 (아이디 + 비밀번호 입력)  = 인증
관리자 메뉴 접근 가능 여부 확인  = 인가
```

A. 인가(Authorization) -> 인증된 사용자에게 특정 리소스나 기능에 대한 권한을 부여, 권한 확인이다. 인증 이후에 수행
B. 로깅(Logging) -> 시스템에서 발생하는 이벤트나 오류를 기록.
**C. 인증 (Authentication)** - 사용자가 자신이 주장하는 사람임을 시스템이 확인하는 과정이다. 아이디/비밀번호, 생체 인식, OTP 등이 인증수단이다.
D. 감사(Auditing) -> 시스템에서 일어난 행위를 추적, 기록하여 보안 검토에 활용하는 과정이다.

- 인증은 당신이 누구인지 확인
- 인가와 자주 혼동되나, 인가는 인증 이후에 무엇을 할 수 있는지 결정

### 5. 구글이나 카카오 계정으로 로그인하는 방식을 주로 무엇이라 부를까요?

- 소셜 로그인은 사용자가 새로운 계정을 직접 만들지 않고, 보유한 소셜 미디어 계정을 활용해 서비스에 로그인 하는 방식이다.
- 내부적으로는 OAuth 2.0 프로토콜을 기반으로 동작한다.
  소셜 로그인 흐름:

```
사용자가 "카카오로 로그인" 버튼 클릭
     ↓
카카오 인증 서버로 이동 (OAuth 요청)
     ↓
사용자가 카카오에서 로그인 + 권한 동의
     ↓
카카오가 인증 코드(Authorization Code) 발급
     ↓
서비스 서버가 코드로 액세스 토큰 교환
     ↓
액세스 토큰으로 사용자 정보 조회
     ↓
로그인 완료 ✅
```

- 별도 회원가입 불필요 -> 가입 허들 감소
- 비밀번호 관리 부담 없음
- 소셜 플랫폼의 보안 인프라 활용이 가능
