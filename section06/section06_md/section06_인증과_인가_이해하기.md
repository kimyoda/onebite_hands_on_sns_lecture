## section06\_인증과인가 이해하기

## 인증, 인가

> **인증 ≠ 인가**
> 비슷해 보이지만 완전히 다른 개념이다.

- 인증 !== 인가
- 인증 (Authentication)
- 사용자의 신원을 확인하는 것
- id, password를 통해 본인의 신원을 증명하는 과정

- 인가
- Authorization
- 인증된 사용자의 권한을 확인하는 과정
- 사용자가 회원정보 같은 관리자용 데이터에 접근하려할 때, 접근 권한을 확인
- 권한이 있다면 접근 허용하는 것

- 인증 -> 로그인 등의 수단으로 사용자의 신원을 확인하는 과정
- 인가 -> 특정 데이터에 접근할 권한을 확인받는 과정

|          | 인증 (Authentication) | 인가 (Authorization)    |
| -------- | --------------------- | ----------------------- |
| **목적** | 신원 확인             | 권한 확인               |
| **질문** | "너 누구야?"          | "너 이거 해도 돼?"      |
| **예시** | 로그인                | 관리자 페이지 접근 허용 |
| **순서** | 먼저 이뤄진다         | 인증 이후에 이뤄진다    |

---

## 인증 방식 두 가지

인증을 처리하는 방식은 **세션 방식**과 **토큰 방식**이 있다.

- 게시물을 삭제하려면, 삭제에 관련된 인증을 거쳐야 한다.
- 게시글을 삭제 할 권한을 확인하는 인가 과정을 거쳐야 한다.

- 서비스의 보안을 위해서 모두 구현해야 한다.

## 1. 세션 방식 (Session)

### 세션 개념

> **서버가 로그인 상태를 직접 기억한다**

로그인에 성공하면 서버가 세션 객체를 만들어 저장,
클라이언트에게 세션 ID만 전달한다.
이후, 요청마다 세션 ID를 확인해 누구인지 파악한다.

### 흐름

```
1. 사용자가 id/password로 로그인 요청
        │
        ▼
2. 서버가 DB에서 회원 정보 조회
        │
        ▼
3. 서버가 세션 객체 생성 후 서버 메모리(또는 DB)에 저장 세션 객체 = { sessionId: "abc123", userId: 1, createdAt: "..." }
        │
        ▼
4. 클라이언트에게 sessionId만 응답
        │
        ▼
5. 클라이언트는 sessionId를 쿠키에 저장
        │
        ▼
6. 이후 모든 요청에 쿠키(sessionId)를 함꼐 전송
        │
        ▼
7. 서버가 sessionId로 저장된 세션 객체 조회 -> 사용자 확인
        │
        ▼
8. 인증된 사용자에게 응답 반환
```

### 장점

- **세밀한 상태 관리 가능**
  - 서버가 세션 객체를 직접 들고 있기 때문에
  - 특정 사용자를 즉시 강제 로그아웃 시킬 수 있다.
    - 예) 계정 도용 감지 시 해당 세션만 삭제하면 된다.

- **보안 제어가 쉽다**
  - 세션을 서버에서 삭제하면 즉시 접근이 차단된다.

### 단점

- **서버 메모리 부담**
  - 사용자가 늘어날수록 세션 객체도 늘어나 메모리 사용량이 증가한다.
  - 동시 접속자가 많은 서비스에는 큰 부담이 될 수 있다.

- **매 요청마다 DB 조회 발생**
  - sessionId를 받을 때마다 DB에서 세션 정보를 찾아야 한다.
  - 요청이 많아질수록 DB 부하가 커진다.

## 2. 토큰 방식 (Token)

### 개념

> **서버가 기억하지 않는다. 토큰 자체다.**

로그인에 성공하면 서버가 토큰을 만들어 클라이언트에 전달한다.
이후 요청마다 토큰을 보내면 서버가 DB 조회 없이 토큰 자체를 검증한다.

### JWT(JSON Web Token)

토큰 방식에 가장 많이 쓰이는 표준형식이 **JWT**다.
JWT는 3가지 파트로 구성된다.

```
eyJhbGciOiJIUzI1NiJ9.eyJ1c2VySWQiOjF9.abc123xyz
      │                     │                 │
   HEADER               PAYLOAD           SIGNATURE
```

**HEADER**
어떤 알고리즘으로 암호화했는지, 토큰 타입 등의 정보를 담는다.
Base64로 인코딩한다.

**PAYLOAD**
사용자 ID, 닉네임, 토큰 만료 시간 등 실제 데이터를 담는다. 인코딩한다
암호화가 아니다.

**SIGNATURE**
Header + Payload를 합친 문자열을 서버만 알고 있는 **비밀키(SecretKey)**로 암호화한다.
서명이 일치하면 서버가 만든 진짜 토큰이다.

### 흐름

```
1. 사용자가 id/password로 로그인 요청
        │
        ▼
2. 서버가 DB에서 회원 정보 조회
        │
        ▼
3. 서버가 JWT 생성 (Header + Payload + Signature) 서버 어디에도 저장하지 않는다.
        │
        ▼
4. 클라이언트에 JWT 응답
        │
        ▼
5. 클라이언트는 JWT를 localStorage 또는 쿠키에 저장
        │
        ▼
6. 이후 모든 요청의 Authorization 헤더에 JWT 포함해 전송 Authorization: Bearer ey...
        │
        ▼
7. 서버가 Signature 검증 -> 유효하면 Payload에 사용자 정보 추출
        │
        ▼
8. 인증된 사용자에게 응답 반환
```

### 장점

- **서버가 상태를 저장하지 않는다(Stateless)**
  DB 조회 없이 토큰 자체만으로 검증, 서버 부하가 적고 확장성이 좋다.

- **다양한 클라이언트 지원**
  웹, 모바일, 서버 등 어디서든 같은 토큰으로 인증할 수 있다.

### 단점

- **토큰 탈취 시 대처가 어렵다**
  세션처럼 서버에서 즉시 삭제할 수 없다. -> **Acess Token + Refresh Token** 조합으로 보완한다.

- **Payload 내용이 노출된다**
  Base84는 암호화가 아니므로 누구나 Payload를 쓸 수 있다.

## Access Token + Refresh Token 패턴

토큰 탈취 무제를 보완하기 위해 실무에서 두 가지 토큰을 함꼐 사용한다.

|               | Access Token         | Refresh Token              |
| ------------- | -------------------- | -------------------------- |
| **역할**      | 실제 API 요청에 사용 | Access Token 재발급에 사용 |
| **유효기간**  | 짧다 (15분 ~ 1시간)  | 길다 (7일 ~ 30일)          |
| **저장 위치** | 메모리 또는 쿠키     | HttpOnly 쿠키 (보안 강화)  |

```
Access Token 만료
        │
        ▼
Refresh Token으로 새 Access Token 재발급 요청
        │
        ▼
서버가 Refresh Token 검증 후 새 Access Token 발급
        │
        ▼
사용자는 로그아웃 없이 계속 사용 가능
```

> Access Token을 짧게 유지해서 탈취 피해를 최소화하고,
> Refresh Token으로 사용자 경험(자동 로그인 유지)을 보완하는 방식이다.

## 📊 세션 vs 토큰 최종 비교

| 항목              | 세션 방식               | 토큰 방식                 |
| ----------------- | ----------------------- | ------------------------- |
| **상태 저장**     | 서버가 저장             | 클라이언트가 저장         |
| **DB 조회**       | 매 요청마다 필요        | 불필요                    |
| **서버 확장**     | 어렵다 (세션 공유 필요) | 쉽다 (Stateless)          |
| **강제 로그아웃** | 즉시 가능               | 어렵다 (만료 전까지 유효) |
| **보안**          | 서버 측 제어 가능       | 탈취 시 대처 어려움       |
| **사용 사례**     | 기업 내부 시스템        | 현대 웹/모바일 서비스     |
