## section06\_인증과인가 이해하기

### 인증 & 인가

- 인증 !== 인가
- 인증 (Authentication)
- 사용자의 신원을 확인하는 것
- id, password를 통해 본인의 신원을 증명하는 과정

- 인가
- Authorization
- 인증된 사용자의 권한을 확인하는 과정
- 사용자가 회원정보 같은 관리자용 데이터에 접근하려할 때, 접근 권한을 확인
- 권한이 있다면 접근 허용하는 것

- 인증 -> 로그인 등의 수단으로 사용자의 신원을 확인하는 과정
- 인가 -> 특정 데이터에 접근할 권한을 확인받는 과정

- 게시물을 삭제하려면, 삭제에 관련된 인증을 거쳐야 한다.
- 게시글을 삭제 할 권한을 확인하는 인가 과정을 거쳐야 한다.

- 서비스의 보안을 위해서 모두 구현해야 한다.

- 인증은 세션 방식과 토큰 방식을 통해 이뤄진다.
- 세션방식
  - 세션 객체를 통해 인증을 처리하는 방식
  - 로그인 -> 회원 정보 조회 -> 세션 객체를 생성(sessionId, userName, 생성정보 등) -> 세션 객체를 저장
  - 세션 Id 응답
  - 쿠키, 스토리지에 보관한다.
  - 다음 요청을 보낼때 세션 Id를 함꼐 보낸다.
  - 세션 개체를 확인한다.
  - 인증된 사용자만 받을 수 있는 응답을 보낸다.
- 장점
  - 세션 객체를 이용해, 사용자들의 상태를 관리할 수 있다
- 단점
  - 메모리를 과하게 사용, 데이터베이스를 자주 조회해야 한다

- 토큰 방식
  - 토큰을 이용해 인증을 처리하는 방식

  - 로그인(id, password)
  - 회원정보 조회
  - 서버가 토큰을 생성한다(아주긴 문자열 형태 데이터), 나만 만들 수 있는 서명 - 시그니처 문자를 생성 -> access token, refresh token을 통해 유효기간을 설정해준다.
  - JWT(JSON Web Token)
    - HEADER -> 암호화 알고리즘, 토큰의 타입 정보를 객체 -> 인코딩
    - PAYLOAD -> 사용자 닉네임, 인증 요청 시간 등을 객체 -> 인코딩
    - SIGNATURE -> 앞서만든 문자열을 합치고 -> 서버만 알고있는 비밀키로 암호화한다.
  - 토큰을 응답
  - 요청을 보낼때 토큰을 보낸다
  - 토큰 검증 과정
  - 문제가 없으면 응답을 보내주는 방식

- 장점
  - 데이터베이스 조회가 필요없다
- 단점
  - 사용자들의 상태를 관리할 수 없다.
  - 토큰 탈취시 대처가 불가능함.
