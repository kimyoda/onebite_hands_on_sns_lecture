### 1. Q1. 여러 사용자가 동시에 '좋아요'를 클릭할 때 발생하는 동시성 문제는 무엇인가요?

- 여러 사용자가 거의 동시에 같은 게시물에 좋아요를 누르면, 각 요청이 같은 시점의 like_count 값을 읽고 각각 업데이트 할 수 있다.

A. 서버 과부하로 인한 서비스 지연
: 서버 지연이 아닌 데이터를 동시에 수정하며 발생하는 카운트 불일치
**B. 좋아요 카운트가 실제보다 적게 업데이트됨** -> 사용자가 동시에 좋아요를 누르면 모두 동일한 초기 카운트를 읽고, 각자 +1 한 값을 저장할 수 있다.
C. 대이터베이스 잠금으로 다른 작업 중단
: 행 장금은 동시성 문제를 해결하기 위한 방법
D. 사용자 인증 오류 발생
: 인증 오류는 로그인 상태나 권한 검증 문제

### 2. Supabase 클라이언트 한계로, 좋아요 기능 행 잠금 구현에 무엇을 사용하나요?

- 좋아요 수를 안전하게 증가/감소 시키면 데이터베이스 레벨에서 게시물 행을 잠근 뒤 업데이트 하는 방식이 필요
- PRC로 함수를 호출하는 방식을 사용한다

A. GraphQL API
: GraphQL은 데이터를 요청하는 API 방식 중 하나이다.
B. RESTful API
: 서버와 통신하는 방식.
**C. RPC, 원격 프로시저 호출** -> `rpc()`를 활용해 해당 SQL 함수를 호출한다.
D. WebSocket 통신
: 실시간 통신에 사용된다.

### 3. 현재 사용자가 특정 게시물을 '좋아요'했는지 여부는 데이터베이스 조인으로 어떻게 확인하나요?

- 게시물 목록이나 상세 정보를 가져올 때, 사용자가 해당 게시물에 좋아요를 눌렀는 지 함꼐 알아야 한다
- `Post` 테이블 자체 `isLiked` 컬럼을 두지않고 로그인한 사용자 기준으로 달라지는 값이다.
- 사용자 ID와 게시물 ID가 일치하는 기록이 있늕디 확인

A. `Post` 테이블의 `isLiked` 컬럼 조회
: `isLiked`는 사용자마다 달라지는 값이다.
**B. `Like` 테이블과 조인 후, 사용자 ID로 필터링** -> 게시물 데이터를 조회할 때 `Like` 테이블을 조인, 로그인한 사용자의 ID로 필터링, 해당 사용자가 게시물을 좋아요했는 지 확인가능
-> 조회 결과에 좋아요 기록이 있으면 `isLiked: true`, 없으면 `isLiked: false`로 가공할 수 있다.
C. `Profile` 테이블에 사용자 좋아요 목록 조회
: `Profile` 테이블은 사용자 정보 저장에 적합
D. 별도의 `LikedPosts` API 호출
: 데이터베이스 조인을 통해 확인하는 방법

### 4.'좋아요' 버튼에 낙관적 업데이트를 적용하는 주된 목적은 무엇인가요?

- 낙권적 업데이트는 서버 응답을 기다리지 않고, 성공할 것이라고 가정한 UI를 먼저 변경하는 방식
- 좋아요 버튼을 누르논 즉시 하트 상태와 좋아요 수를 먼저 바꾸고, 서버 요청이 실패하면 이전 상태로 롤백한다

A. 서버 부하 감소
: 낙관적 업데이트는 UI를 먼저 바꾸는 전략. 서버 부하 감소가 주된 목적은 아니다
B. 데이터 일관성 완벽 보장
: 낙관적 업데이트는 임시로 서버 상태와 클라이언트 상태가 다를 수 있다. 실패 시 롤백이나 재검증이 필요하다
**C. 즉각적인 UI 피드백 제공** -> 낙권적 업데이트은 사용자 경험 개선이다. 좋아요 상태와 카운트를 변경하여 이전 데이터를 저장하고 롤백하는 처리 필요
D. 보안 취약점 개선
: 보안은 인증, 권한 검사, RLS, 서버/RPC 검 증으로 처리

### 5. `Like` 테이블에 `SELECT` RLS 정책만 필요한 주된 이유는 무엇인가요?

- RLS는 Row Level Security 약자,
- 테이블 각 행에 대해 어떤 사용자가 읽거나 쓸 수 있는지 제한하는 기능
- 검증 로직이 들어 있는 RPC 함수에 처리하도록 만들 수 있다

A. `Like` 테이블은 읽기 전용
: `Like` 테이블은 읽기 전용이 아니다. INSERT, DELETE가 필요하고 이 작업을 RPC로 활용한다
**B. 쓰기 작업은 RPC 함수로 처리**-> `Like` 테이블에 접근해서 처리하지 않고, `SECURITY DEFINER` 권한을 가진 RPC 함수에서 안전하게 처리한다. SELECT 권한만 열어둬서 대응한다
C. 모든 사용자에게 `Like` 데이터 공개
: 사용자 행동 정보이므로 무조건 모두에게 공개하면 안된다.
D. `Like` 데이터는 보안이 중요하지 않다
: 사용자의 활동 기록이다.
