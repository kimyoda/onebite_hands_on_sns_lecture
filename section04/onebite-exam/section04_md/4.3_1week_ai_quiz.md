Q1. TailwindCSS의 유틸리티 우선(Utility First)접근 방식

- **C. 작고 단일 목적의 유틸리티 클래스를 마크업에 바로 적용한다.**
- TailwindCSS는 유틸리티 우선(Utility-First) CSS 프레임워크다.
- 기존 CSS 방식처럼 컴포넌트에 이름을 붙이고 별도 파일에 스타일 자성하는게 아닌 `text-center`, `bg-blue-500`, `mt-4`처럼 하나의 역할만 하는 작은 클래스들을 HTML에 직접 조합해 UI를 만든다.
- 번들 크기도 최적화 된다.

A. 각 컴포넌트에 대한 커스텀 CSS를 작성한다 -> 전통적인 BEM, SCSS방식이다. Tailwind는 커스텀 CSS 작성을 최소화하는 것이 목표다.
B. 미리 만들어진 UI 컴포넌트와 테마를 사용한다 -> Bootstrap, Material UI 같은 컴포넌트 기반 라이브러리 특징이다. Tailwind는 완성된 컴포넌트를 제공하지 않는다.
D. 자바스크립트 객체를 기반으로 스타일링한다. -> 이는 CSS-in-JS 방식(Styled Components, Emotion)의 특징이다.

Q2. 단일 페이지 애플리케이션(SPA)에서 React Router의 주요 목적

**정답: C - 서로 다른 뷰 간의 선언적 탐색을 가능하게 한다**

- SPA(Single Page Application)는 페이지 전체를 새로 로드하지 않고, JavaScript로 화면을 동적으로 교체한다.
- React Router는 URL 경로(path)에 따라 어떤 컴포넌트를 렌더링할지 선언적으로 정의할 수 있게 해주는 라우팅 라이브러리다.
- 브라우저의 주소창은 바뀌지만 실제 페이지 이동은 발생하지 않는다.

A. 애플리케이션의 전역상태를 관리한다. -> Redux, Zustand, Context API 같은 상태 관리 도구 역할이다.
B. 비동기 데이터 가져오기를 처리한다. -> Tanstack Query(ReactQuery), SWR, `useEffect + fetch` 등의 역할이다.
C. 컴포넌트 렌더링 성능을 최적화한다 -> 이는 `React.memo`, `useMemo`, `useCallback` 등 React 최적화 도구의 역할이다.

Q3. React 애플리케이션 Zustand의 주요용도

**정답: C - 효율적인 글로벌 상태 관리**

- Zustand는 React를 위한 경량 전역 상태 관리 라이브러리다
- Redux에 비해 코드가 훨씬 적고, Context API보다 불필요한 리렌더링을 방지하는데 유리하다.
- 스토어(store)를 만들고 컴포넌트에서 필요한 상태만 구독(subscribe)하는 방식으로 동작한다.

A. 재사용 가능한 UI 컴포넌트를 구축한다 -> shadcn/ui, Radix UI, Headless UI 등 UI 컴포넌트 라이브러리 역할이다
B. 서버 측 데이터 가져오기를 관리한다 -> TanStack Query, SWR의 역할이다.
C. 복잡한 UI 애니메이션을 정의한다 -> Framer Motion, React Spring 같은 애니메이션 라이브러리 역할이다.

Q4. 서버상태(Sever State)와 클라이언트 상태(Client State)의 구분

**정답: C - 서버 상태는 원격으로 저장, 클라이언트 상태는 로컬로 저장된다**

A. 서버 상태는 항상 동기적이다 -> 서버 상태는 오히려 비동기 네트워크 요청으로 가져온다.
B. 서버 상태는 영구적이다 -> 서버 상태도 삭제되거나 변경될 수 있어 항상 영구적이지는 않는다.
D. 서버 상태는 항상 읽기 전용이다 -> 서버 상태도 POST/PUT/DELETE로 변경(쓰기)가능하다.

Q5. 서버 데이터를 수정하는데 사용되는 Tanstack Query 훅

\*\*정답: C - `useMutation`

- 서버 상태 관리를 위한 라이브러리, 두 가지 훅이 있다.
- `useQuery`: 데이터 읽기(조회), GET 요청, 목록 불러오기
- `useMutation`: 데이터 변경(생성/수정/삭제), POST, PUT, DELETE 요청

- `useMutation`은 요청 성공 후 `invalidateQueries`를 호출해 관련 캐시를 무효화, 최신 데이터를 다시 불러오는 패턴과 함꼐 자주 사용한다.

A. `useQuery` -> 데이터를 조회하는 훅이다. 서버 데이터를 변경하는 용도가 아니다.
B. `useQueryClient` -> TanstackQuery의 캐시에 직접 접근하기 위한 훅이다. 데이터 변경 요청 자체를 담당하지 않는다.
D. `useUpdater` -> Tanstack Query에 존재하지 않는 훅이다.
