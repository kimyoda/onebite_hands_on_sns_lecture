# 4_1. 서버 상태 관리란?

### 서버 상태 관리와 TanStack Query

- 앱의 테마 데이터, 세션 정보 데이터 -> 전역 상태 분류

- 특정 <input>의 입력 값 -> 지역 상태(Local State)

- 게시글 리스트 API 요청 -> 비동기 API 요청에 관한건 어디로 분류?
  - isLoading(로딩 상태), status(성공, 실패 유무), error(에러 객체) 등
  - 모든 데이터를 전역상태 데이터를 직접 관리하려고 하면 굉장히 많은 state값이 존재하고, 각각의 액션 함수도 많이 생겨야 하는 수고로움이 생긴다.
  - 대부분의 서버들이 API를 하나만 쓰지 않기에, 요청과 관련된 데이터는 전역 상태로 관리하기에는 무리가 있다.
  - 지역 상태는 여러 컴포넌트가 이용할 수 없고, 코드가 매우 복잡해지는 건 전역 상태와 같다.

- 서버 상태라는 별도의 유형으로 관리한다(Server State)
- TanStack Query라는 가장 대표적인 서버 상태 관리 라이브러리를 사용한다.
  - 비동기 상태를 매우 편리하게 관리할 수 있게 하는 다양한 기능을 제공한다.

### 2. TanStack Query란?

> API요청과 관련돈 모든 상태(로딩 ,에러, 데이터, 캐싱)을 자동으로 관리해주는 서버 상태 관리 라이브러리

- TanStack Query를 쓰면 아래처럼 단 한줄로 API 요청과 관련된 상태를 얻을 수 있다.

```tsx
export default function Component() {
  const { data, isLoading, error } = useQuery(...)

  if (error) return <div>오류가 발생했습니다!</div>
  if (isLoading) return <div>로딩 중 입니다 ...</div>
  return <div>{data}</div>
}
```

- `useState`로 isLoading, error, data를 하나씩 만들고 관리할 필요가 없다.

### 3. TanStack Query의 핵심 기능

**자동 캐싱 (Caching)**

- 한 번 가져온 데이터는 메모리에 저장해둔다. 같은 데이터를 요청하면 API를 다시 호출하지 않고 캐시에서 즉시 반환한다.
  - 첫 번째 요청 → API 서버 호출 → 데이터 캐시에 저장
  - 두 번째 요청 → 캐시에서 즉시 반환 (API 호출 없음)

**자동 백그라운드 갱신 (Background Refecthing)**

- 캐시된 데이터가 오래됐다고 판단되면 백그라운드에서 자동으로 최신 데이터를 가져온다.
- 사용자가 다른 탭을 갔다가 돌아오는 순간에도 자동으로 갱신된다(`refetchOnWindowsFocus`)

- 로딩/에러 상태 자동 관리
  - API 요청의 상태를 자동으로 추적해서 제공한다.

```ts
const {
  data,        // 성공 시 받아온 데이터
  isLoading,   // 최초 로딩 중 여부
  isFetching,  // 백그라운드 갱신 중 여부
  isError,     // 에러 발생 여부
  error,       // 에러 객체
  isSuccess,   // 성공 여부
} = useQuery(...)
```

**staleTime과 gcTime**

- staleTime
  — 데이터를 "신선하다"고 볼 시간, 이 시간 안에는 캐시 데이터를 그대로 사용 (재요청 없음)

- gcTime
  — 캐시를 메모리에서 완전히 삭제하기 전까지 보관하는 시간, (구버전 이름: cacheTime)

\*\*자동재시도(Retry)

- API 요청이 실패하면 자동으로 재시도한다.
- 기본값은 3회이고 횟수를 직접 설정할 수도 있다.

```ts
useQuery({
  queryKey: ["todos"],
  queryFn: fetchTodos,
  retry: 3, // 실패 시 최대 3번 재시도
});
```

### 4. useQuery vs useMutation

TanStack Query의 두 핵심 Hook이다.

- useQuery
  - 용도: 데이터 읽기
  - HTTP메서드: GET
  - 자동실행: 컴포넌트 마운트 시 자동실행
  - 캐싱: O
  - 게시글 목록 불러오기

- useMutation
  - 용도: 데이터 변경
  - HTTP 메서드: POST/PUT/PATCH/DELETE
  - 자동실행: 수동으로 직접 실행
  - 캐싱: X
  - 게시글 작성, 수정, 삭제

### 5. Query Key - 캐시 핵심

- TanStack Query는 **Query Key**를 기준으로 캐시를 관리한다.
- 같은 캐시를 공유한다.

```ts
// 문자열 단독
useQuery({ queryKey: ['todos'], ... })

// 배열 — 파라미터가 있을 때
useQuery({ queryKey: ['todos', 1], ... })      // id가 1인 todo
useQuery({ queryKey: ['todos', { isDone: true }], ... })  // 완료된 todo만
```

- Key가 다르면 별개의 캐시로 취급된다.
- 필터, id, 페이지 같은 값이 바뀌면 Key도 함께 바꿔줘야 최신 데이터를 정확하게 가져온다.
