# 캐싱 매커니즘 이해하기

### 핵심확인

> "TanStack Query는 사용자에게 로딩 화면을 최대한 덜 보여주려 하는 것"

- 전통적인 `useEffect + fetch` 방식은 컴포넌트가 마운트 될때마다 무조건 서버에 요청을 보낸다
- 사용자는 매번 로딩 스피너를 봐야한다.

- TanStackQuery는 다르다.
- **이전에 가져온 데이터를 캐시에 저장, 다음번엔 캐시를 먼저 보여준다**
- 백그라운드에서 조용히 최신 데이터를 가져오고, 도착하면 자연스럽게 교체한다.
- 사용자 입장에선 즉각적인 응답처럼 느껴진다

---

### 📦 캐시의 5가지 상태 흐름

```
[서버 요청 시작]
      ↓
  fetching  →  fresh  →  stale  →  inactive  →  deleted
                ↑_______________|
          (stale 상태에서 특정 조건이 되면 다시 fetching)
```

| 상태         | 의미                          | 특징                                       |
| ------------ | ----------------------------- | ------------------------------------------ |
| **fetching** | 서버에서 데이터를 불러오는 중 | 네트워크 요청 진행 중                      |
| **fresh**    | 데이터가 신선한 상태          | 리페칭 조건이 와도 요청 안 함              |
| **stale**    | 데이터가 오래된 상태          | 조건이 되면 백그라운드 리페칭              |
| **inactive** | 구독 컴포넌트가 없는 상태     | gcTime 타이머 시작, 데이터는 메모리에 존재 |
| **deleted**  | 메모리에서 완전 제거된 상태   | gcTime 만료 후 가비지 컬렉션               |

---

### 🔄 캐싱 흐름 예시

```
1. 사용자가 Todo 상세 페이지(/todos/1) 진입
    -> ["todos", 1] 캐시 없ㄷ음 -> fetching 시작 -> isLoading = true -> 로딩화면

2. 데이터 도착 -> 캐시에 저장
    -> staleTime이 0이면 즉시 stale 상태로 전환
    -> staleTime이 5000이면 5초간 fresh 유지

3. 사용자가 목록 페이지로 이동 (상세 컴포넌트 unmount)
    -> ["todos", 1] 캐시의 구독자가 사라짐
    -> inactive 상태로 전환
    -> gcTime 타이머 시작(5분)

4-A. gcTime 내에 다시 /todos/1 진입
    -> 캐시가 살아있음 -> 캐시 데이터 즉시 보여줌 (로딩 없음)
    -> stale 상태면 백그라운드에서 조용히 리페칭

4-B. gcTime 초과 후 /todos/1 진입
    -> 캐시가 삭제됨 -> fetching 시작 -> 다시 로딩화면
```

> inactive가 됐다고 바로 삭제하지 않는 이유다. 사용자가 돌아왔을 때 로딩 없이 즉시 보여주기 위해 gcTimwe동안 캐시를 보존한다.
