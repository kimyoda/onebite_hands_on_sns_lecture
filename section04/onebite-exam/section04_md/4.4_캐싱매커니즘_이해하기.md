# 캐싱 매커니즘 이해하기

> TanStack Query의 주요요소: **사용자에게 로딩화면을 최대한 덜 보여준다**
> 네트워크요청이 완료되기 전에도 이전에 캐싱해둔 데이터를 화면에 보여주고,
> 백그라운드에서 최신 데이터를 조용히 가져온다.
> 사용자 입장에서는 거의 항상 즉각적인 응답처럼 느껴진다.

### TanstackQuery 캐시(fetching, fresh, stale)까지만 다룬다

```
fetching → fresh → stale → inactive → deleted
              ↑__________________|
              (stale 상태에서 특정 조건이 되면 다시 fetching)
```

| 상태         | 의미                                        |
| ------------ | ------------------------------------------- |
| **fetching** | 서버에서 데이터를 불러오는 중               |
| **fresh**    | 데이터가 신선한 상태 (리페칭 안 함)         |
| **stale**    | 데이터가 오래된 상태 (조건이 되면 리페칭)   |
| **inactive** | 이 캐시를 구독하는 컴포넌트가 없는 상태     |
| **deleted**  | `gctime`이 지나 캐시에서 완전히 제거된 상태 |

### fetching

- 데이터를 서버에서 불러오는 중인 상태.

* stale에서 다시 리페칭 (데이터를 다시 불러올 때)
  1. Mount: 이 캐시 데이터를 사용하는 컴포넌트가 마운트 되었을 때
  2. WindowFocus: 사용자가 이 탭에 다시 돌아왔을 때
  3. Reconnect: 인터넷 연결이 끊어졌다가 다시 연결되었을 때
  4. Interval: 특정 시간을 주기로

### fresh

- 데이터가 신선한 상태일 때, `staleTime`이 지나지 않은 동안 유지된다.
  - fresh 상태에서 데이터가 충분히 최신이라고 판단하기 때문이다.
  - 캐시에 있는 데이터가 충분히 최신이라고 판단하기 때문이다.
  - **사용자 입장에서 로딩없이 즉시 데이터가 보인다** TanStack Query의 중요한 점

> `staleTime`의 기본값은 9이다. 데이터를 받는 즉시 stale로 전환된다.

### stale

- 데이터가 상한 상태
  - stale 상태에서도 캐시 데이터는 화면에 보여준다.
  - 리페칭 조건이 충족되면 백그라운드에서 조용히 새 데이터를 가져온다
  - 새 데이터가 도착하면 화면을 교체한다.

> 사용자는 로딩 스피너 없이 기존 데이터를 보다가, 새 데이터로 자연스럽게 교체되는 경험을 한다.

### inactive

- 이 캐시를 **구독(사용)하는 컴포넌트가 하나도 없는** 상태.
- Todo 목록 페이지에 상세 페이지로 이동, 컴포넌트가 마운트되며 `["todos]` 캐시를 구독하던 컴포넌트가 사라진다. 캐시는 inactive 상태로 전한된다.
  - 데이터는 아직 메모리에 살아있다.
  - 리페칭은 발생하지 않는다 (구독자가 없으니 보여줄 곳도 없다)
  - inactive 상태에서 `gctime`타이머가 시작된다.

> inactive가 되었다고 바로 삭제하지 않는다. 사용자가 다시 목록 페이지로 돌아왔을 때, 캐시가 살아있으면 **로딩 없이 즉시 보여줄 수 있다** 로딩을 최대한 덜 보여준다

```
gcTime 기본값: 5분 (300,000ms)
→ 아무도 쓰지 않는 캐시를 5분간 메모리에 보존했다가, 그 이후에 삭제한다.
```

### deleted

- `gcTime` 지나 캐시가 **메모리에서 완전히 제거된** 상태
  - inactive 상태로 `gcTime`(기본 5분)이 경과하면 가비지 컬렉션(GC)대상이 되어 삭제된다.
  - 삭제된 이후에 해당 쿼리를 다시 사용, 캐시가 없는것과 같으므로 **처음부터 fetching이 시작**된다(`isLoading = true`)

> staleTime, gcTime 비교

|옵션| 역할|기본값|
|`staleTime`| fresh->state 전환까지의 시간 | 0(즉시 stale)|
|`gcTime`|inactive->deleted 전환까지의 시간 | 설정값|

- `staleTime`은 "언제부터 리페칭을 허용할지"를 제어
- `gcTime` "언제 메모리에서 지울지"를 제어
- `gcTime`은 항상 `staleTime`보다 길게 설정하는 것이 일반적이다. (stale이 되기도 전에 캐시가 삭제되면 의미가 없기 때문이다)
