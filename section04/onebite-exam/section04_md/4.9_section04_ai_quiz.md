1. 비동기 API 요청 데이터 관리에 가장 적합한 상태 유형은 무엇인가?

- 프론트엔드에서 상태(State)는 4가지로 분류된다

| 상태유형    | 설명                                                    |
| ----------- | ------------------------------------------------------- |
| 로컬 상태   | 특정 컴포넌트 내부에서만 사용되는 상태                  |
| 글로벌 상태 | 여러 컴포넌트에서 공유되는 애플리케이션 전역 상태       |
| 서버 상태   | 서버에서 가져오는 비동기 데이터 (로딩, 에러, 캐싱 포함) |
| UI 상태     | 모달 열림/닫힘, 다크모드 등 순수하게 UI에만 관련된 상태 |

- 비동기 API 요청은 "데이터를 저장하는 것" 이상의 복잡한 라이프사이클을 가진다.

- 요청 중인가? (로딩 상태)
- 요청이 실패했는가? (에러 상태)
- 데이터가 최신인가? (stale 상태)
- 언제 다시 가져와야 하는가? (캐싱, refetch)

이런 복잡한 관리를 위해 **서버 상태(Sever State)** 라는 개념이 등장했고, TanStack Query를 활용하였다.

**정답: C - 서버 상태**

- API 요청 데이터는 로딩, 에러, 캐싱, refetch 등 복잡한 라이프사이클을 가지고, 이를 효율적으로 처리하기 위해 **서버 상태** 개념이 등장했다.
- 로컬/글로벌 상태만으로 무관한 순수한 클라이언트 상태다.

- A. 로컬 상태 -> `useState`로 관리하는 컴포넌트 내부 상태이다. API 데이터를 저장할 수 있으나, 로딩/에러 처리, 캐싱, 자동 refetch등을 직접 구현해야 한다.
- B. 글로벌 상태 -> Redux, Zustand 등으로 관리하는 전역 상태이다. 컴포넌트에서 공유는 되나, 서버 데이터의 신선도(staleness), 캐싱, 백그라운드 업데이트 등을 기본적으로 지원하지 않는다.
- D. UI 상태 -> 모달 표시, 선택된 앱 등 UI의 시각적 상태
