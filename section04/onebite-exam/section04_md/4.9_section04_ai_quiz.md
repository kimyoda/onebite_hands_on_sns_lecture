# Ai quiz

### section04_quiz

1. 비동기 API 요청 데이터 관리에 가장 적합한 상태 유형은 무엇인가?

- 프론트엔드에서 상태(State)는 4가지로 분류된다

| 상태유형    | 설명                                                    |
| ----------- | ------------------------------------------------------- |
| 로컬 상태   | 특정 컴포넌트 내부에서만 사용되는 상태                  |
| 글로벌 상태 | 여러 컴포넌트에서 공유되는 애플리케이션 전역 상태       |
| 서버 상태   | 서버에서 가져오는 비동기 데이터 (로딩, 에러, 캐싱 포함) |
| UI 상태     | 모달 열림/닫힘, 다크모드 등 순수하게 UI에만 관련된 상태 |

- 비동기 API 요청은 "데이터를 저장하는 것" 이상의 복잡한 라이프사이클을 가진다.

- 요청 중인가? (로딩 상태)
- 요청이 실패했는가? (에러 상태)
- 데이터가 최신인가? (stale 상태)
- 언제 다시 가져와야 하는가? (캐싱, refetch)

이런 복잡한 관리를 위해 **서버 상태(Sever State)** 라는 개념이 등장했고, TanStack Query를 활용하였다.

**정답: C - 서버 상태**

- API 요청 데이터는 로딩, 에러, 캐싱, refetch 등 복잡한 라이프사이클을 가지고, 이를 효율적으로 처리하기 위해 **서버 상태** 개념이 등장했다.
- 로컬/글로벌 상태만으로 무관한 순수한 클라이언트 상태다.

- A. 로컬 상태 -> `useState`로 관리하는 컴포넌트 내부 상태이다. API 데이터를 저장할 수 있으나, 로딩/에러 처리, 캐싱, 자동 refetch등을 직접 구현해야 한다.
- B. 글로벌 상태 -> Redux, Zustand 등으로 관리하는 전역 상태이다. 컴포넌트에서 공유는 되나, 서버 데이터의 신선도(staleness), 캐싱, 백그라운드 업데이트 등을 기본적으로 지원하지 않는다.
- D. UI 상태 -> 모달 표시, 선택된 앱 등 UI의 시각적 상태

2. `gcTime`이 만료되면 비활성(inactive) 캐시 데이터는 어떻게 되나?

```
데이터 페치 완료
     ↓
  Fresh 상태 (staleTime 이내 → 신선한 데이터)
     ↓ staleTime 경과
  Stale 상태 (오래된 데이터, 조건 충족 시 refetch)
     ↓ 컴포넌트 언마운트 (구독자 없음)
  Inactive 상태 (비활성 캐시로 전환)
     ↓ gcTime 경과
  메모리에서 완전 삭제 (Garbage Collection)
```

- `staleTime`: 데이터가 신선(fresh)한 상태로 유지되는 시간
- `gcTime`: 비활성 캐시 데이터가 메모리에 유지되는 시간
- `gcTime`이 만료되면 더 이상 사용되지 않는 캐시 데이터를 가비지 컬렉션하여 메모리에서 완전히 제거한다.

**정답: C - 메모리에서 삭제된다.** -> `gcTime`은 비활성 캐시 데이터의 수명을 결정한다. 만료되면 해당 데이터는 메모리에서 완전히 제거, 같은 쿼리를 실행하면 처음부터 새로 서버에 데이터를 가져온다.

- A. 다시 Fresh 상태가 된다. -> `gcTime`은 캐시 유지 시간, 데이터를 Fresh로 리세하는 기능이 없다. Fresh상태는 `staleTime`으로 제어한다.
- B. 자동으로 Refetch 된다. -> 자동 Refetch는 컴포넌트가 마운드 되거나, 윈도우 포커스가 돌아오는 등의 트리거에 의해 발생한다.
- D. Stale 상태가 된다. -> Stale상태는 `staleTme`이 지난 후 상태이다.

3. 백엔드 데이터 수정(생성/업데이트/삭제)에 사용되는 훅은 무엇인가?

- TanStack Query는 데이터와 **읽기(read)**와 **쓰기(raed)**을 명확하게 분리한다.
  | 훅 | 용도 | HTTP 메서드|
  |---|---|---|
  |`useQuery` | 데이터 조회 (읽기) | GET |
  | useMutation` | 데이터 변경 (쓰기) | POST, PUT, PATCH, DELETE |

- `useMutation`은 다음과 같은 콜백을 제공한다.

```js
const mutation = useMutation({
  mutationFn: (newTodo) => axios.post("/todos/", newTodo),
  onMutate: (variables) => {
    /* 요청 직전 실행 */
  },
  onSuccess: (data) => {
    /* 성공 시 실행 */
  },
  onError: (error, variables, context) => {
    /* 실패 시 실행*/
  },
  onSettled: () => {
    /* 성공, 실패 무관 항상 실행 */
  },
});
```

**정답: C - `useMutation`** -> 서버 데이터의 생성, 수정, 삭제와 같은 CUD(Create, Update, Delete) 작업을 위한 훅이다. `useQuery`가 읽기(R)를 담당, `useMutation`은 쓰기 (CUD)을 담당한다.

- A. `useQuery` -> 데이터 조회(GET) 전용 훅이다. 컴포넌트 마운트 시 자동으로 실행, 캐싱과 자동 refetch를 담당한다. 데이터 변경에 적합하지 않는다.
- B. `useState` -> React 기본 훅, 컴포넌트 내 로컬 상태를 곤라힌다. 서버 통신 기능이 없다.
- D. `useEffect` -> React 기본 훅, 사이드 이펙트를 처리한다. API 호출을 직접 넣을 수 있지만, 로딩/에러 상태, 캐싱, 재시도 등을 모두 직접 구현해야 한다.

4. 낙관적 업데이트 실패시, 캐시를 원래 상태로 되돌리려면 무엇을 사용해야 하나?

- 낙관적 업데이트(Optimistic Update)란, 서버 응답을 기다리지 않고 UI를 먼저 업데이트 하는 기법이다.
  - UX가 즉각적으로 반응하여 빠르게 느껴진다
  - 서버 요청 실패 시 UI를 원래 상태로 롤백해야 한다.

```js
useMutation({
  mutationFn: updateTodo,

  // 1단계: 요청 전, 이전 데이터를 저장하고 UI를 먼저 업데이트
  onMutate: async (newTodo) => {
    await queryClient.cancelQueries({ queryKey: ["todos"] });
    const previousTodos = queryClient.getQueryData(["todos"]); // 이전 데이터 저장
    queryClient.setQueryData(["todos"], (old) => [...old, newTodo]); // 낙관적 업데이트
    return { previousTodos }; // context로 반환
  },

  // 2단계: 실패 시, context에서 이전 데이터를 꺼내 롤백
  onError: (err, newTodo, context) => {
    queryClient.setQueryData(["todos"], context.previousTodos); // 롤백
  },

  // 3단계: 성공, 실패 무관 최종 동기화
  onSettled: () => {
    queryClient.invaildateQueries({ queryKey: ["todos"] });
  },
});
```

**정답: B - `onMutate` 반환 값과 `onError`** -> `onMutate`에서 이전 데이터를 저장해 `context`로 반환, `onError`에서 그 `context`를 이용해 캐시를 복구한다. 낙관적 업데이트 롤백의 핵심패턴이다.

- `onMutate` 이전 데이터를 저장, `context`로 반환 / `onError`: `context`에서 이전 데이터를 꺼내 `setQueryData`로 캐시 복구한다.

- A.. `onSuccess` 핸들러 -> 요청 성공 시에만 실행된다. 실패 상황에서 호출되지 않으므로 롤백로직을 담기에 부적절하다.
- C. `onSettled` 핸들러 -> 성공과 실패 모두에서 실행된다. `invalidateQueries`로 최종 동기화에 사용, 롤백 로직을 여기에 넣으면 성공 시에도 실행되는 문제가 발생한다.
- D. `invalidateQueries` -> 특정 쿼리 캐시를 무효화하여 refetch를 트리거한다. 롤백보다는 서버로부터 최신 데이터를 다시 가져오기에 적합하다.

5. TanStack Query에 캐시 정규화의 목적은 무엇인가?

- **캐시 정규화(Cache Normalization)**은 동일한 데이터가 여러 쿼리 캐시에 중복 저장되는 문제를 해결하는 방법이다.
- 정규화 없이(중복캐시):

```
캐시 키: ['todos']          → [{ id: 1, title: '밥 먹기' }, { id: 2, title: '운동' }]
캐시 키: ['todos', 1]       → { id: 1, title: '밥 먹기' }  ← 중복!
캐시 키: ['user', 'todos']  → [{ id: 1, title: '밥 먹기' }]  ← 중복!
```

- id: 1의 title이 바뀌면, 세 군데를 모두 업데이트 해야한다.

정규화 후(단일 출처):

```
캐시 키: ['todos', 1]  → { id: 1, title: '밥 먹기' }  ← 단 하나의 원본
다른 캐시들은 id만 참조  → id: 1 → 항상 최신 데이터 반환
```

- 하나만 업데이트 후 모든 곳에 반영된다.

**정답: B - 중복 캐시 데이터 제거 및 효율적 관리** -> 정규화의 핵심 목적이다. 동일 데이터의 중복 저장을 없애고, 동일 데이터의 중복 저장을 없앤다. 단일 항목을 기준으로 캐시를 관리하여 업데이트를 단순화한다.
-> 캐시 정규화는 동일 데이터(특정 사용자, 특정 아이템)를 **하나의 캐시 항목으로 단일화**하여 중복을 제거한다.
-> 이를 통해 데이터 업데이트가 단순해지고, 모든 관련 쿼리에서 항상 일관된 최신 데이터를 볼 수 있다.

A. 데이터 캐시 속도 향상 -> 정규화의 직접적인 목적이 아니다. 정규화는 데이터를 개별항목으로 분리, 일관성과 효율성에 초점이 있다.
C. 네트워크 요청 수 최소화 -> 캐싱 전략(`staleTime`, `gcTime`)의 목적에 가깝다. 정규화는 이미 캐시된 데이터의 구조를 다루는 개념이다.
D. 캐시 데이터의 자동 새로 고침 -> 이는 `staleTime`, `refectchOnWindowFocus` 등의 역할이다. 정규화와는 별개의 개념이다.
